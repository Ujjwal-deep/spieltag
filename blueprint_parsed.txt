📌 Project Overview
Build a Bundesliga match outcome predictor using a 4-model ensemble (ELO + Logistic Regression + XGBoost + LLM). Data is fetched from free APIs, stored in SQLite, and presented through a React frontend backed by a FastAPI server. The system retrains every 4–6 weeks and monitors for concept drift.

3.2  Cold Start Handling (Newly Promoted Teams)
Promoted teams (e.g. Holstein Kiel entering Bundesliga) have no top-flight history. The pipeline must detect and handle this before building any feature row, otherwise features will be missing or misleadingly filled with zeros.
ELO — Promoted Team Starting Rating
Do NOT start promoted teams at the default 1500. Use their Bundesliga 2 finish to set a realistic initial ELO:
# Bottom of Bundesliga ≈ ELO 1380
# BL2 champion gets close to the weakest BL1 side
promoted_elo = 1350 + (bl2_final_position_from_bottom * 10)
# e.g. BL2 champion (18th from bottom) → 1350 + 180 = 1530
H2H Features — Tiered Fallback
H2H Bundesliga matches available
Strategy
5 or more
Use actual H2H stats — full confidence
2 to 4
Use available H2H but multiply by (n / 5) to weight down confidence
0 to 1
Drop H2H features entirely. Fill with Bundesliga average home win rate (~43%)
xG & Form — Bundesliga 2 Fallback
If a team has < 3 Bundesliga matches (i.e. newly promoted), fall back to their Bundesliga 2 stats from last season
Apply a promotion discount factor of 0.85 to all BL2 stats — BL2 is weaker, so raw stats inflate
Use the is_home_promoted and home_bl2_xg_avg5 columns to signal this in the feature row
Understat tracks Bundesliga 2, so this data is available without any extra API
LLM Model — Best Tool for Cold Start
The LLM has no minimum data requirement. Explicitly include in the prompt that a team is newly promoted, their BL2 finishing stats, and any known signings. Claude reasons qualitatively where statistical models cannot.
3.3  No-Leakage Rule
When computing rolling features for a match on date D, ONLY use matches with date < D. Use pandas shift(1) to ensure the current match is never included in its own features.
PHASE 4: Model Training
⏱ ~4 days
Model 1 — ELO Rating System
Start all teams at 1500
K-factor: 32 for regular season, 40 for high-stakes matches
Update ratings after every result using standard ELO formula
ELO naturally handles long-term strength — no retraining needed, it's always live
Output: win/draw/loss probabilities derived from rating difference
Model 2 — Logistic Regression
Features: form_pts, elo_diff, h2h stats, home/away record
Multinomial (3 classes: home win, draw, away win)
Apply recency weighting: sample_weight = exp(-0.001 * days_ago)
Regularization: C=0.1 (L2), prevents overfitting on limited data
Output: [P(home win), P(draw), P(away win)]
Model 3 — XGBoost
Full feature set: all 15 features from Phase 3
n_estimators=300, max_depth=4, learning_rate=0.05
Use sample_weight for recency (same decay as LogReg)
Tune with 5-fold time-series cross-validation (TimeSeriesSplit) — never shuffle
Output: [P(home win), P(draw), P(away win)]
Model 4 — LLM Reasoning (Claude API)
Prompt Claude with a structured JSON summary of both teams. Example prompt structure:
You are a Bundesliga analyst. Given the following pre-match data, predict
the outcome probabilities as JSON: {"home_win": float, "draw": float, "away_win": float}
Home team: {name}, ELO: {elo}, Last 5: {form}, xG avg: {xg}
Away team: {name}, ELO: {elo}, Last 5: {form}, xG avg: {xg}
Context: {injuries, manager changes, streak, H2H}
Respond ONLY with the JSON object. No explanation.

🤖 Notes for Claude Agent
When working on this project as an agent, follow these rules:
Always check that rolling features use shift(1) or date < match_date — never allow the current match's stats into its own feature row.
Never shuffle training data when using TimeSeriesSplit — older matches must always be in train, newer in validation.
Cache all LLM calls to the DB before matchday — never call Claude API on page load or per-request.
When adding new features, add them to the features table in Phase 3 of this document and retrain all models.
API-Football is ONLY for lineups and injuries — never use it for possession or shots. FBref (soccerdata) handles those with no rate limit.
FBref via soccerdata: always set data_dir to a local cache folder and add a ~1s delay between requests to avoid soft-blocking.
ELO must be recomputed in chronological order — always sort by date ASC before processing.
For newly promoted teams: check is_home_promoted / is_away_promoted before building features. Route to BL2 fallback stats and apply 0.85 discount factor.
For H2H features: check h2h_data_quality. If 'none', replace H2H features with Bundesliga average home win rate (0.43). If 'partial', weight down by (n / 5).
Promoted teams ELO: never start at 1500. Compute from BL2 finish: 1350 + (position_from_bottom × 10).
When the Brier score degrades >15% from baseline, log a warning and schedule a retrain job.
Ensemble weights are in ensemble.py as constants — update them after each quarterly review.
LLM model: use claude-sonnet-4-20250514, max_tokens=300, temperature=0 for deterministic predictions.
For cold start matches (promoted team, thin H2H), always increase the LLM ensemble weight to 0.40 and reduce XGBoost to 0.20 — statistical models are least reliable here.
